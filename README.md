# Health Communication Analyst Agent

A RAG-based, agentic analyst workflow that converts thesis-derived health-video transcript, audio, and model evidence into structured communication-quality reports with schema validation, QA guardrails, and evaluation traces.

This repository is a software-engineered extension of a larger multimodal thesis research project on health-related YouTube videos. The original thesis pipeline collected and modeled approximately 10,000 public health videos using transcript, audio, duration, metadata, interpretable features, and engagement-normalized labels. This repository focuses on the downstream analyst layer: transforming exported thesis artifacts into auditable, evidence-grounded reports that explain how a video’s communication signals compare with higher-engagement reference patterns.

The project is designed to demonstrate a workflow pattern relevant to scientific AI systems: domain-specific retrieval, structured outputs, human-interpretable reporting, automated quality checks, and traceable evaluation logs.

---

## Project overview

The Health Communication Analyst Agent analyzes a selected health-video example using three evidence sources:

1. **Transcript evidence**
   Cleaned transcript excerpts are retrieved to support metric-level interpretations.

2. **Model and feature evidence**
   Thesis-derived interpretable metrics, audio summaries, model scores, and class labels are loaded from exported artifacts rather than generated by the LLM.

3. **Domain/rubric evidence**
   Metric definitions, communication-quality guidelines, and report templates are retrieved from a local knowledge base.

The workflow produces:

* a structured communication-quality report;
* a scientist-facing email draft;
* metric-level evidence references;
* QA validation results;
* local evaluation traces;
* optional LangGraph/LangSmith-compatible workflow instrumentation.

## Why this project exists

The original thesis work was developed primarily in Google Colab because the upstream workflows required GPU-backed processing for embeddings, model training, and multimodal analysis. However, the downstream interpretation layer benefits from a cleaner software architecture.

This repository separates the **scientific artifact generation layer** from the **AI analyst/reporting layer**.

The thesis pipeline produces artifacts such as:

* cleaned transcripts;
* normalized interpretable feature scores;
* audio summary metrics;
* model prediction outputs;
* engagement class labels;
* high-engagement reference statistics.

This repository consumes those artifacts and turns them into structured, auditable reports.

The goal is not to rerun the full thesis pipeline. The goal is to demonstrate a reusable pattern for AI-assisted scientific workflows where outputs must be grounded, validated, and understandable to non-technical scientific stakeholders.

## GenAI workflow focus

This project emphasizes several areas relevant to applied scientific AI and agentic workflow development:

### Agentic workflow orchestration

The pipeline is organized as a controlled multi-step workflow:

```text
load thesis artifacts
        ↓
retrieve transcript and rubric evidence
        ↓
generate structured analyst report
        ↓
run QA guardrails
        ↓
revise or finalize
        ↓
write outputs and evaluation trace
```

The workflow is intentionally constrained. The agent does not autonomously invent facts or metric values. Instead, it routes between artifact loading, retrieval, report generation, validation, and output writing.

### Retrieval-augmented generation

The RAG layer retrieves two kinds of evidence:

* **rubric evidence** from the project knowledge base;
* **video-specific evidence** from transcript excerpts and model-derived artifacts.

This allows the analyst report to connect metric-level interpretations back to explicit source material.

### Structured outputs

Reports are represented with Pydantic schemas to ensure predictable output structure. The schema enforces required report sections, metric-level interpretations, evidence fields, limitations, and stakeholder-facing summaries.

### QA guardrails

The workflow includes deterministic validation checks for:

* expected metric count;
* missing or unexpected metric names;
* evidence coverage;
* unsupported claims;
* overly technical stakeholder language;
* report completeness.

This mirrors scientific and regulated workflow needs where generated text should be checked before being shared with human reviewers.

### Evaluation traces

Each run writes a local JSONL evaluation trace containing run metadata, validation status, metric coverage, evidence coverage, warnings, and errors. This provides lightweight observability even without external services.

The project is also structured to support optional LangSmith-style tracing and review workflows.

The full thesis datasets are not included in this repository.

Private thesis files are stored locally under:

```text

```

Small demo-ready artifacts are generated under:

```text

```

The public/demo subset is created from private thesis CSVs using:

```bash
python scripts/build_public_demo_subset.py \
  --features data/private/interpretable_feature_dataset_v3.csv \
  --videos data/private/healthVideoData_preprocessed.csv \
  --output-dir data/public \
  --n 100 \
  --label 0 \
  --seed 42 \
  --target-baseline data/reference/class1_high_like_metric_means.json
```

The conversion process creates:

```text

├── video_inputs/
├── metric_bundles/
├── transcripts/
├── manifest.jsonl
└── README.md
```

The generated demo subset is intentionally small and is used to demonstrate the analyst workflow without exposing the full thesis dataset.

## Metric framework

The report uses 15 thesis-derived communication metrics:

```text
first_person_rate
second_person_address_rate
third_person_informational_rate
structured_explanation_score
engagement_request_score
formality_score
medical_vocabulary_rate
youth_style_rate
emotional_intensity
readability_grade_level
lexical_diversity_mtld
sentiment_polarity
pitch_variability
loudness_variability
voice_clarity_hnr
```

These metrics are loaded from exported thesis artifacts. They are not generated by the LLM.

Metric values are compared against a high-engagement reference profile derived from class-1 videos. This allows the workflow to identify candidate improvement directions, such as whether a lower-engagement video has less structured explanation, fewer engagement prompts, lower medical vocabulary rate, or different audio delivery patterns than the higher-engagement reference group.

The comparison is descriptive rather than causal. The system does not claim that changing any single metric will guarantee higher engagement.

## Installation

On macOS, the system may not provide a `python` command by default. Install Python through Homebrew if needed:

```bash
brew install python
```

Then create and activate a virtual environment:

```bash
python3 -m venv .venv
source .venv/bin/activate

pip install -r requirements.txt
```

For Windows PowerShell:

```powershell
python -m venv .venv
.venv\Scripts\Activate.ps1

pip install -r requirements.txt
```

## Design principles

### 1. The LLM is not the source of truth

Metric values, model scores, class labels, and audio summaries are loaded from thesis artifacts. The analyst workflow may interpret and summarize them, but it should not invent them.

### 2. Outputs should be structured and validated

Scientific workflows benefit from predictable outputs. Pydantic schemas and deterministic guardrails reduce the risk of incomplete or malformed reports.

### 3. Evidence should be retrievable and inspectable

Metric interpretations should be grounded in retrieved transcript excerpts, model artifacts, or rubric context. Reports should make the evidence trail visible to reviewers.

### 4. Recommendations should be cautious

The system identifies communication patterns associated with higher relative engagement in the thesis framework. It does not claim causality, medical accuracy, or guaranteed engagement improvement.

### 5. The workflow should support human review

The generated report and email draft are intended to assist a human reviewer, not replace one. The architecture is designed for human-in-the-loop review and future feedback collection.

---

## Current limitations

This repository demonstrates the downstream analyst workflow, not the full thesis modeling system.

Limitations include:

* the full 10,000-video dataset is not included;
* the demo subset is small and selected for workflow demonstration;
* metric comparisons are descriptive, not causal;
* the system does not evaluate medical accuracy;
* transcript retrieval quality depends on transcript quality;
* audio metrics are summary features and may be affected by recording conditions;
* stakeholder recommendations should be reviewed by a human before use.

---

## Relevance to scientific AI workflows

Although this project uses public health-video data, the architecture is designed to transfer to more formal scientific workflows.

The same pattern could be adapted to settings where an AI system must:

* load structured and unstructured scientific artifacts;
* retrieve internal guidelines, templates, or reference materials;
* generate structured outputs;
* check outputs against deterministic criteria;
* maintain traceable evaluation logs;
* support human review and revision;
* communicate clearly with scientist users.

This makes the project a practical demonstration of RAG-based, agentic workflow design for scientific communication analysis.
